# 🎯JOB자 - 신입 구직자를 위한 AI 커리어 파트너

<img width="1071" height="570" alt="JOB 자의 사본 (1)" src="https://github.com/user-attachments/assets/2f192d94-b1f6-403a-b6a2-055ab80407fd" />


> 2025년 현재, 국내 채용 시장은 빠르게 변화하고 있으며, 특히 **신입 구직자에게 혹독한 빙하기**가 찾아왔습니다.  
> 채용 공고 수는 줄고, 실무 경험을 요구하는 ‘중고 신입’을 선호하는 기업은 늘고 있습니다.  
>  
> 하지만 신입 구직자에게 주어진 정보는 방대하고 파편화되어 있으며, **지금 시장이 원하는 역량이 무엇인지**, **자신이 어떤 부분에서 부족한지**, **어떤 공고에 지원해야 유리한지**를 스스로 분석하고 판단해야 합니다.  
>
> **JOB자**는 이러한 문제의식에서 출발했습니다.  
> 신입 구직자의 입장에서 **채용 시장을 읽고**, **본인에게 유리한 기회를 찾고**, **취업까지의 경로를 설계**할 수 있도록 도와주는
> **AI 기반 커리어 파트너**입니다.


#### **JOB자**는 신입 구직자가 혼자서 헤매지 않도록

✔️ 실시간 시장 분석으로 지금 당장 필요한 역량을 알려주고  

✔️ 이력서 기반 개인 맞춤 공고 추천과  

✔️ 갭 분석 → 로드맵 → 액션 플랜까지 제안하여  

구직자가 **최단 경로로 취업**할 수 있도록 돕습니다.

### 1. 임베딩 모델

| 항목           | 설명                                                                                     |
|----------------|--------------------------------------------------------------------------------------------|
| **모델**       | `intfloat/multilingual-e5-large` (SentenceTransformer 기반)                               |
| **입력 구조**  | 채용공고: `(주요업무×2 + 자격요건×2 + 우대사항 + 기술스택)`  <br> 사용자: 기본정보 + 희망직무 + 경력 + 스킬 + 경험 + 자격증 |
| **처리 방식**  | 통합 텍스트를 하나로 합쳐 임베딩 후 Cosine 유사도 계산                                   |
| **저장 방식**  | PostgreSQL + `pgvector` 기반 Vector DB 구축                                               |
| **평가지표**   | 평균 유사도, 유사도 분산, 중복 추천 공고 수 등                                           |

### 2. 클러스터링 모델

| 항목           | 설명                                                                                     |
|----------------|--------------------------------------------------------------------------------------------|
| **모델**       | `KMeans` (클러스터 수 K=200)                                                              |
| **목적**       | 기존 직무 분류 체계의 한계를 보완하고, 의미 기반 직무 재분류 수행                         |
| **활용 방식**  | 직무 리매핑, 공고 필터링, 트렌드 분석 등에 활용                                           |
| **평가지표**   | Intra/Inter-cluster 거리, Silhouette Score, Calinski-Harabasz, Davies-Bouldin             |
| **성과 요약**  | 응집도 향상 (Intra 감소), 군집 간 구분도 향상 (Inter 증가)                                |

### 3. LLM 기반 재랭킹

| 항목           | 설명                                                                                     |
|----------------|--------------------------------------------------------------------------------------------|
| **모델**       | `llama-4-maverick`                                                                         |
| **입력 구조**  | 사용자 이력서 요약 + Top-30 공고 전문 요약                                                |
| **출력 결과**  | Top-5 추천 공고 + 추천 사유 + 부족 역량 및 개선 방향 제안                                 |
| **기술 구성**  | Prompt 튜닝 기반 자연어 추론, 조건 부족 시 보완방안도 함께 제시                            |
| **역할 요약**  | 유사도 기반 1차 필터링 결과를 재정렬하고, 설명 가능한 AI 추천으로 연결                    |

## 1. 테스트 환경 

| 항목 | 내용 |
|------|------|
| **프레임워크** | FastAPI (비동기 지원) |
| **비동기 클라이언트** | `httpx.AsyncClient` 사용 (FastAPI 서버와의 내부 API 통신) |
| **서버 구조** | MCP 서버 ↔ 내부 FastAPI 서버 호출 ↔ PostgreSQL 연결  |
| **의존 시스템** | PostgreSQL DB, JWT 인증, 외부 FastAPI API 서버 |
| **호스트** | `localhost`, 포트 `8001` (→ 내부 통신 또는 로컬 테스트 환경) |
| **API 통신 구조** | 대부분 `GET`, 일부 `POST`, `PUT` 비동기 방식으로 API 호출 처리 |
| **도구 수** | `AVAILABLE_TOOLS` 기준 20개 이상의 기능이 정의됨 (추천, 이력서 수정, 갭 분석, 시각화 등) |

## 2. 테스트 결과

| 항목 | 평가 내용 |
|------|-----------|
| **응답 시간** | `httpx.AsyncClient` 기반 비동기 요청 처리 구조 → 실 응답 속도는 평균 1~2초 |
| **기능 호출 성공률** | 대부분의 API는 예외처리 되어 있으며, 중복/에러 상태를 명확히 반환하여 안정적인 흐름 제어 가능 |
| **기능 테스트 범위** | 20개 이상의 도구에 대해 등록, 수정, 조회, 분석, 추천 등 실 기능 호출 가능 (ex: `resume`, `skills`, `job_posts`) |
| **에러 처리** | `try-except`와 `HTTPException` 기반으로 네트워크/인증/입력값 오류를 구분하여 처리함 |
| **확장성** | 각 도구는 `endpoint_mapping`으로 라우팅되어 있어 API 변경 없이 유연한 기능 추가 및 수정 가능 |
| **일관성** | `get_my_resume`, `update_resume`, `job_recommendation` 등 주요 기능은 사전 조회 및 중복 검사 로직을 통해 일관된 응답 유지 |

## 3. 성능 개선 필요 사항
| 항목 | 평가 내용 |
|------|-----------|
| **응답 포맷 일관성 확보** | 도구마다 반환 형식이 상이하여 상위 시스템 연동 시 파싱 부담이 있음 → 공통 응답 스키마 적용 또는 최소한의 포맷 통일 필요 |
| **LLM 응답 안정성 개선** | `job_recommendation`, `gap_analysis` 등 일부 도구는 LLM 기반 응답에 의존하며 결과 포맷이 유동적임 → 포맷 검증 및 fallback 로직 필요 |
| **기초 성능 모니터링 도입** | MCP 서버의 요청 처리 시간, 성공률, 오류 로그 등에 대한 정량적 수집 구조가 없음 → 간단한 로그 기반 모니터링 체계 도입 필요 |

</details>

  <summary> 서동겸 회고록 </summary>
  
  ### 잘된 점
  - 이번 프로젝트에서는 데이터 수집 및 처리 자동화를 성공적으로 구축했습니다. Apache Airflow를 활용하여 DAG을 구성하고, 특히 전처리 과정에서 로컬 LLM 서버를 사용하여 비용을 절감한 점은 큰 성과였습니다.

  - 과거에는 CSS 선택자나 XPath를 직접 사용하여 웹 데이터를 수집했기 때문에, 웹사이트의 작은 DOM 구조 변경에도 크롤러가 쉽게 중단되는 문제가 잦았습니다. 이를 해결하기 위해 웹 페이지 내부에 숨겨진 JSON 형태의 데이터를 찾아 파싱하는 방식으로 수집 로직을 변경했습니다. 그 결과, 클래스명 변경과 같은 잦은 웹사이트 업데이트에도 안정적으로 데이터를 수집할 수 있는 견고한 크롤러를 구현할 수 있었습니다.

  - 데이터 수집 방식도 크게 개선했습니다. 기존의 단일 for 루프 방식에서 벗어나 비동기 및 병렬 처리를 도입하여 작업 시간을 획기적으로 단축했습니다. 물론, 대상 서버에 과부하를 주지 않도록 비동기 요청 수를 적절히 조절하고, 트래픽이 적은 새벽 시간대에 작업을 실행하는 등 안정성도 함께 고려했습니다.

### 개선할 점
  - 기술적 측면에서, 데이터 수집 과정의 한계점이 존재합니다. 채용 공고 본문이 텍스트가 아닌 단일 이미지로 제공되는 경우가 종종 있어 현재 로직으로는 데이터를 추출할 수 없습니다. 향후 OCR 등을 도입하여 이미지 내 텍스트도 유연하게 처리할 수 있도록 수집 로직을 고도화해야 합니다. 또한, 프로젝트 규모가 커짐에 따라 Airflow DAG의 개수가 늘어나고 Task 간 의존성이 복잡해졌습니다. 더 복잡한 파이프라인을 효율적으로 관리하고 전체 워크플로우를 조율하는 오케스트레이션 능력을 향상시키기 위한 추가 학습이 필요합니다.

  - 협업 및 리더십 측면에서, 개인 프로젝트와 팀 단위 협업은 차원이 다름을 느꼈습니다. 특히 팀장으로서 아이디어 회의 주도, 팀원 간 의견 조율, 명확한 역할 배분, 그리고 CI/CD 환경 구축 및 운영 등에서 많은 미숙함을 보였습니다. 앞으로는 체계적인 협업 방식을 도입하여 이러한 부족한 점들을 개선해야 합니다. 예를 들어, Figma나 Slack과 같은 협업 툴을 적극적으로 활용하고, 효율적인 회의를 위한 명확한 규칙을 세우며, 섬세한 스케줄 관리를 통해 프로젝트 진행 상황을 투명하게 공유하는 노력이 필요하다고 느꼈습니다.

### 교훈
  - 이번 프로젝트를 통해 **"현실의 데이터는 결코 예쁘지 않다"** 는 점을 절실히 깨달았습니다. 교육용으로 잘 정제된 데이터와 달리, 실제 필드에서 마주하는 데이터는 수집 단계부터 예측 불가능한 문제들로 가득했습니다. 데이터 수집뿐만 아니라, 수집된 데이터를 목적에 맞게 가공하고 정제하는 과정의 중요성을 명확히 인지하게 되었습니다. 실제 프로젝트는 이론처럼 순탄하게 진행되지 않는다는 현실을 받아들이고, 문제 해결 능력을 기르는 것이 중요하다는 교훈을 얻었습니다.

### 잘된 점
#### 1. 기능 중심 구조화

기능별로 컴포넌트를 체계적으로 분리했습니다. 예를 들어, Sidebar.jsx는 전체 네비게이션, JobCardPreview.jsx는 채용 공고 목록, AiRecsPreviewCard.jsx는 AI 추천 요약, TodoList.jsx는 할 일 관리 등 각각의 파일에 역할이 분명합니다. 이러한 설계 덕분에 특정 기능을 수정하거나 디버깅할 때 다른 코드에 미치는 영향을 최소화할 수 있었습니다

#### 2. Context API를 활용한 효율적인 전역 상태 관리

애플리케이션 전반에서 사용되는 데이터들을 기능별로 분리하여 중앙에서 관리한 것이 성공적이었습니다.
* AuthContext.jsx는 사용자의 로그인 상태와 프로필 정보를 전역적으로 제공하여 어떤 컴포넌트에서든 인증 상태에 따른 UI를 쉽게 구현할 수 있게 했습니다.
* JobNamesContext.jsx는 여러 필터와 드롭다운에서 공통으로 사용하는 직무 목록을 한 번만 불러와 제공함으로써 불필요한 API 호출을 막고 데이터의 일관성을 유지했습니다.
* RecommendationContext.jsx는 AI 추천 데이터와 로딩 상태를 관리하여, AiRecsPreviewCard.jsx와 같은 UI 컴포넌트가 데이터 호출 로직과 완전히 분리될 수 있도록 했습니다. 

#### 3. API 로직의 모듈화를 통한 관심사 분리

서버와 통신하는 모든 로직을 api 디렉토리에 mcp.js, trendApi.js 와 같이 기능별 파일로 모듈화한 것은 코드의 가독성과 유지보수성을 크게 향상시켰습니다. 덕분에 UI 컴포넌트들은 데이터가 어떻게 오는지 신경 쓸 필요 없이 무엇을 보여줄지에만 집중할 수 있었고, API가 변경되더라도 디렉토리 내부만 수정하면 되어 유연하게 대처할 수 있었습니다.

### 개선할 점
#### 1. 공통 UI 컴포넌트의 부재와 스타일 중복 

프로젝트 전반에 걸쳐 Card, Button, Modal 등 유사한 UI 요소들이 각 파일 내에서 styled-components로 개별적으로 정의되어 사용되고 있습니다. 예를 들어, AiJobRecommendation.jsx의 JobCard와 AiRecsPreviewCard.jsx의 HoverCard는 기본적인 카드 형태를 공유하지만 별도로 구현되었습니다. 공통 UI 컴포넌트를 모아두는 디렉토리(ex: src/components/common)를 만들고 재사용 가능한 형태로 설계했다면, 코드 중복을 줄이고 전체적인 디자인의 일관성을 더욱 효과적으로 유지할 수 있었을 것입니다.

#### 2. useState와 useEffect를 통한 서버 상태 관리의 복잡성 

데이터를 불러오는 로직이 대부분 useState와 useEffect 훅에 의존하고 있습니다. 이는 maincontent.jsx처럼 여러 종류의 비동기 데이터를 다뤄야 하는 컴포넌트에서 수많은 상태(loading, error, data 등)와 useEffect 의존성 배열 관리의 복잡성을 초래했습니다. 서버 상태 관리 라이브러리를 도입했다면, 데이터 캐싱, 재요청, 로딩 및 에러 상태 처리를 훨씬 선언적이고 간결한 코드로 해결할 수 있었을 것입니다.

### 교훈
#### 1.  리팩토링의 중요성 

프로젝트 초기, maincontent.jsx는 페이지 렌더링을 총괄하는 거대한 지휘자 역할을 했습니다. 빠른 프로토타이핑에는 유리했지만, 기능이 추가될수록 수많은 상태와 복잡한 조건부 렌더링 로직이 집중되면서 유지보수의 어려움이라는 가장 큰 실수에 직면했습니다. 이 문제를 해결하기 위해 AiRecsPreviewCard, TodoPreview와 같은 기능들을 독립된 컴포넌트로 분리하고, UserDataContext, RoadmapContext 등 전역 상태를 도입하여 maincontent의 부담을 점진적으로 덜어냈습니다. 이 과정은 "초기 설계의 실수를 인정하고, 프로젝트를 위해 과감하게 리팩토링하는 것이 장기적으로 훨씬 이득"이라는 값진 교훈을 주었습니다.

#### 2. API 중복 호출 방지를 통한 성능 향상 

프로젝트가 복잡해지면서 여러 컴포넌트가 동일한 데이터를 필요로 하는 상황이 발생했습니다. 예를 들어, JobCardFiltered.jsx의 필터 드롭다운과 MyProfile.jsx의 관심 직무 선택 UI 모두 전체 직무 목록이 필요했습니다. 초기에는 각 컴포넌트가 개별적으로 직무 목록 API를 호출하여, 페이지 로드 시 동일한 API가 여러 번 중복으로 호출되는 비효율이 발생했습니다. 이는 느린 초기 로딩 속도의 주범이었고, 사용자 경험을 저해하는 심각한 문제였습니다.
이 문제를 해결하기 위해 JobNamesContext.jsx를 도입한 것은 성능을 극적으로 향상시킨 최고의 결정이었습니다. 이 Context는 애플리케이션이 시작될 때 직무 목록 API를 단 한 번만 호출하고, 그 결과를 내부 상태에 저장(캐싱)합니다. 이후 직무 목록이 필요한 모든 컴포넌트(JobSelector.jsx 등)는 API를 직접 호출하는 대신 Context로부터 데이터를 즉시 공급받습니다. 이 간단한 구조 변경만으로 불필요한 네트워크 트래픽을 없애고, 페이지 전환과 초기 로딩 속도를 크게 개선할 수 있었습니다. 이는 데이터를 한 곳에서 관리하고 나눠주는 것이 얼마나 중요한가를 깨닫게 된 핵심적인 교훈입니다.
